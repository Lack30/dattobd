# dattobd 内部工作机制

## 简介

dattobd 模块通过实现 [写时复制 (Copy-on-Write, COW)](http://en.wikipedia.org/wiki/Copy-on-write) 系统来管理快照和追踪变更。当首次为块设备初始化时，内核模块在驱动器上创建一个 COW 数据存储文件。内核模块在块级别拦截所有写入操作，在写入完成之前将即将被更改的数据复制到快照存储中。通过这种方式，即使正在发生写入操作，模块也能维护文件系统的一致且完整的快照。快照数据由内核模块使用位于磁盘 COW 文件开头的索引进行管理和访问。这种实现使我们的写时复制系统能够可靠地维护给定驱动器在块级别的完整时间点映像，这意味着它比其他实现我们目标的方法更可靠、更一致。


### 术语
* **扇区 (Sector)**: 块设备上固定大小的用户可访问数据块。内核将其定义为 512 字节。
* **块 (Block)**: 连续扇区组成的块。在我们的情况下，为 4096 字节，即 8 个扇区。
* **区段 (Section)**: COW 管理器使用的基本大小单位。长度为 4096 个扇区。

## COW 文件概述
COW 快照文件可以处于以下三种状态之一：
* 休眠 (Dormant)
* 活动 (Active)
* 未验证 (Unverified)

以及以下两种模式之一：
* 快照模式 (Snapshot)
* 增量模式 (Incremental)

快照设备在任何给定时间都恰好处于一种模式和一种状态。例如，快照设备可以处于增量模式和活动状态，或快照模式和休眠状态。但是，快照设备不能同时处于活动状态和未验证状态，也不能同时处于快照模式和增量模式。

### 快照设备状态
快照设备根据其追踪的块设备的状态在状态之间转换。快照设备大部分时间都处于"活动"状态。在此状态下，快照设备已完全初始化，所有内存数据结构都处于活动状态。只要块设备已挂载，快照就会保持在活动模式。

当块设备被卸载时（无论是用户操作还是计算机正常关机期间），块设备将转换为"休眠"状态。简而言之，"休眠"状态是操作系统可以安全卸载块设备的状态。当 dattobd 模块正在追踪的块设备收到卸载系统调用时，内核模块首先将该快照设备的所有内存数据结构写入 COW 文件以备后用，关闭该文件的所有打开文件描述符，并关闭该块设备的所有处理线程。此时，操作系统可以卸载该卷。快照将保持在休眠模式，直到磁盘再次重新挂载，此时它将转换回活动状态。此状态很重要，因为它允许快照和追踪在卸载期间持续存在。

最后一种状态是"未验证"状态。此状态允许我们的快照设备和追踪在重启期间保持一致和持久。在早期启动过程中，内核模块可以在被追踪的驱动器本身被操作系统挂载之前，使用所有驱动器的数据结构进行初始化。这允许内核模块在系统启动期间驱动器挂载时立即开始追踪。这确保模块捕获被追踪块设备的每一次写入，从而使我们能够为每个卷维护 100% 一致和准确的变更数据。一旦操作系统挂载了驱动器本身，内核模块就会将它们转换为活动状态。

### 快照设备模式
快照设备可以处于增量模式或快照模式。

在快照模式下，磁盘上的 COW 文件维护一个磁盘上的块缓存。此缓存大约占用驱动器总大小的 10%，用于在备份过程中实现写时复制机制。当快照文件处于快照模式时，每当发生写入时，所有将被更新的块首先被复制到磁盘缓存中，然后真正的写入才会执行。这允许内核模块在创建快照的时间点无缝呈现文件系统的一致且准确的视图。

相比之下，在增量模式下，磁盘上只保留 COW 索引。这释放了磁盘空间供文件系统其余部分使用。写入在内存中追踪，并按需定期同步到索引文件。在此模式下，驱动程序不呈现可读的快照设备，只是追踪 COW 索引中哪些块已更改。因此，不需要快照模式期间使用的更昂贵的磁盘块缓存，因为内核模块不需要文件系统中的原始数据。索引稍后可以由 `update-img` 等工具（参见 `utils/` 目录）使用，仅复制快照之间的更改块。在备份之间，驱动程序大部分时间应处于增量模式。

## COW 文件组件
COW 数据存储文件由三个主要部分组成：
* 头部 (Header)
* 索引 (Index)
* COW 数据存储 (COW datastore)

头部和索引始终作为文件的一部分存在。当模块将磁盘转换为快照模式时分配数据存储，当磁盘回到增量模式时释放数据存储。

### COW 文件头部布局

COW 文件头部占据文件的前 4096 字节，分布如下：

* 0x0000 - 0x0003 : 文件签名
* 0x0004 - 0x0007 : 标志
* 0x0008 - 0x000F : 文件指针位置
* 0x0010 - 0x0017 : 为快照模式 COW 文件分配的磁盘空间量
* 0x0018 - 0x001F : 快照序列 ID
* 0x0020 - 0x002F : 快照系列的 UUID
* 0x0030 - 0x0037 : 头部格式版本
* 0x0038 - 0x003F : 快照以来更改的块数
* 0x0040 - 0xFFFF : 空

头部的大部分留空，以备将来使用和对齐考虑。

### 索引

索引是记录块设备哪些区段自上次快照以来已更改的记录。当设备处于增量模式时，此记录会持续更新。

### COW 数据存储
在快照模式下，COW 文件的这一部分用于保存文件系统在拍摄快照时更新的区段的旧版本。通过将这些旧版本存储在 COW 数据存储中，内核模块可以在快照启动的时间点呈现文件系统的一致视图。默认情况下，此临时数据存储分配卷总空间的 10%。希望快照时间不会太长，以至于在快照过程完成之前空间就耗尽。在增量模式下，COW 文件的这一部分被释放并归还给文件系统。

## 内存布局

在内存中，索引是一个 `cow_section` 结构体数组。`cow_section` 结构体如下所示：
```c
struct cow_section{
    char has_data; //如果此区段有映射（在文件上或内存中），则为非零值
    unsigned long usage; //追踪此区段使用频率的计数器
    uint64_t *mappings; //块地址数组
};
```

每个结构体对应磁盘上的一个区段。当块设备首次初始化为被追踪设备时，`has_data` 和 `usage` 都设置为 0，`mappings` 初始化为 null。当区段变为使用中时，`has_data` 设置为 1，`usage` 递增，`mappings` 指向数据本身，数据存储在内存中直到需要刷新到磁盘。为了存储映射数据，模块为每个被追踪的块设备分配一定量的内存。当块设备的内存映射缓冲区填满时，缓冲区会刷新到磁盘，并相应更新 cow_section 结构体。一个重要的注意事项是，每次刷新映射缓冲区时，所有 `usage` 字段都设置为 0。

### 启动期间重新加载数据

当系统在重启或上电事件期间启动时，dattobd 模块在早期启动过程中，即 initramfs 阶段，接管块设备。重新加载过程与初始化过程大致相同，关键区别在于元数据是从头部重新加载而不是生成的。此外，所有区段的 `has_data` 标志都设置为所有区段，表示驱动程序应假设所有区段都有数据同步到磁盘且没有空洞。这要求 dattobd 在卷挂载之前，在 initramfs 中加载到内核中。

### 将数据刷新到磁盘

当内存缓存填满时，驱动程序使用每个 `cow_section` 结构体的 `usage` 字段来计算哪些区段使用最少。然后，基于使用最少的区段不一定需要在内存中的假设，驱动程序将较少使用的一半区段刷新到磁盘。此操作后，缓存半满，仅包含较多使用的区段。然后所有 `usage` 字段重置。请注意，即使 `cow_section` 的缓冲区已刷新到磁盘，其 `has_data` 仍然保持设置。
