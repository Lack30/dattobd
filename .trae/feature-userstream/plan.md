# dattobd 改造方案：磁盘改动传输到用户态进程

## 需求分析

**当前机制**：
- 磁盘写操作被 BIO 拦截后，原始数据写入 COW 文件（Header + Index + Datastore）
- 用户通过 `/dev/datto0` 读取快照视图

**目标机制**：
- 磁盘写操作被拦截后，数据直接传输到用户态进程
- 不再使用 COW 文件存储
- 用户态进程负责处理/存储/转发这些数据

---

## 技术方案分析

### 方案一：Netlink Socket 扩展

**原理**：扩展现有 Netlink 通信机制，用于传输块数据

**架构**：
```
内核空间                          用户空间
┌─────────────┐                  ┌─────────────┐
│ BIO 拦截    │                  │ 用户态进程   │
│ (tracer.c)  │                  │             │
└──────┬──────┘                  └──────▲──────┘
       │                                │
       ▼                                │
┌─────────────┐    Netlink Socket       │
│ 数据封装    │ ──────────────────────► │
│ (块地址+数据)│   (nlmsg_unicast)       │
└─────────────┘                  ┌─────────────┐
                                 │ 数据处理    │
                                 │ 存储/转发   │
                                 └─────────────┘
```

**优点**：
- 复用现有 Netlink 框架
- 实现相对简单
- 支持双向通信

**缺点**：
- Netlink 单次传输有限制（通常 64KB）
- 大块数据需要分片
- 性能开销较大

**适用场景**：低频写操作、小数据量

---

### 方案二：字符设备 + mmap（推荐）

**原理**：创建专用字符设备，用户态进程通过 mmap 共享内存区域

**架构**：
```
内核空间                          用户空间
┌─────────────┐                  ┌─────────────┐
│ BIO 拦截    │                  │ 用户态进程   │
│ (tracer.c)  │                  │             │
└──────┬──────┘                  └──────▲──────┘
       │                                │
       ▼                                │
┌─────────────┐    mmap 共享内存        │
│ 环形缓冲区  │ ◄══════════════════════►│
│ (Ring Buffer)│                       │
└─────────────┘                  ┌─────────────┐
                                 │ 读取缓冲区  │
                                 │ 处理数据    │
                                 └─────────────┘
```

**优点**：
- 零拷贝，性能高
- 支持大数据量传输
- 环形缓冲区避免内存分配开销

**缺点**：
- 需要同步机制（自旋锁/信号量）
- 实现复杂度较高

**适用场景**：高频写操作、大数据量

---

### 方案三：perf/ring-buffer 机制

**原理**：使用 Linux 内核的 perf ring buffer 机制

**架构**：
```
内核空间                          用户空间
┌─────────────┐                  ┌─────────────┐
│ BIO 拦截    │                  │ 用户态进程   │
│ (tracer.c)  │                  │             │
└──────┬──────┘                  └──────▲──────┘
       │                                │
       ▼                                │
┌─────────────┐    perf_event_open      │
│ Perf Ring   │ ◄══════════════════════►│
│ Buffer      │                         │
└─────────────┘                  ┌─────────────┐
                                 │ poll() 等待 │
                                 │ 读取事件    │
                                 └─────────────┘
```

**优点**：
- 内核原生支持
- 高性能
- 支持事件通知

**缺点**：
- 需要学习 perf 子系统 API
- 可能受 perf 配置限制

---

### 方案四：relayfs（推荐）

**原理**：使用 Linux relayfs 子系统，专为内核到用户态大数据传输设计

**架构**：
```
内核空间                          用户空间
┌─────────────┐                  ┌─────────────┐
│ BIO 拦截    │                  │ 用户态进程   │
│ (tracer.c)  │                  │             │
└──────┬──────┘                  └──────▲──────┘
       │                                │
       ▼                                │
┌─────────────┐    relayfs 文件         │
│ relay channel│ ────────────────────► │
│ (per-CPU)   │   /sys/kernel/debug/   │
└─────────────┘   relay/dattobd_data   │
                                 ┌─────────────┐
                                 │ read() 读取 │
                                 │ 处理数据    │
                                 └─────────────┘
```

**优点**：
- 内核原生支持，专为大数据传输设计
- per-CPU 缓冲区，无锁竞争
- 支持子缓冲区切换
- 自动处理溢出

**缺点**：
- 需要挂载 debugfs
- 单向传输（内核→用户态）

**适用场景**：高频、大数据量、单向传输

## 数据格式设计

### 传输数据结构

```c
// 每个数据块的传输格式
struct block_change_event {
    uint64_t block_addr;        // 块地址 (块号)
    uint64_t timestamp;         // 时间戳 (纳秒)
    uint32_t data_len;          // 数据长度 (通常 4096)
    uint32_t seq_num;           // 序列号 (用于排序和去重)
    uint8_t  operation;         // 操作类型: 0=写, 1=其他
    uint8_t  reserved[7];       // 保留字段
    uint8_t  data[];            // 变长数据 (实际块内容)
};
```

### 控制消息格式

```c
// Netlink 控制消息
enum control_msg_type {
    MSG_START_STREAM = 20,      // 开始数据流
    MSG_STOP_STREAM = 21,       // 停止数据流
    MSG_GET_STATUS = 22,        // 获取状态
    MSG_CONFIGURE = 23,         // 配置参数
    MSG_ACK = 24,               // 确认响应
};

struct control_msg {
    uint32_t msg_type;          // 消息类型
    uint32_t minor;             // 设备号
    uint64_t param1;            // 参数1
    uint64_t param2;            // 参数2
};
```

## 改造步骤

### 第一阶段：内核模块改造

1. **新增 relayfs 支持**
   - 文件: `src/relay_channel.c/h` (新增)
   - 功能: 初始化 relayfs 通道、写入数据、清理资源

2. **修改 tracer.c**
   - 修改 `snap_trace_bio()` 函数
   - 将 COW 写入改为 relayfs 写入
   - 保留原有的块地址计算逻辑

3. **修改 netlink_handlers.c**
   - 新增控制消息处理
   - 支持启动/停止数据流

### 第二阶段：用户态进程开发

1. **数据接收模块**
   - 多线程读取 relayfs
   - poll() 等待数据

2. **数据处理模块**
   - 解析数据格式
   - 去重、排序、校验

3. **存储/转发模块**
   - 本地文件存储
   - 网络传输（可选）

### 第三阶段：测试与优化

1. **性能测试**
   - 吞吐量测试
   - 延迟测试
   - CPU 开销测试

2. **可靠性测试**
   - 进程崩溃恢复
   - 缓冲区溢出处理
   - 数据完整性校验

## 关键技术点

### 1. 缓冲区大小配置

```c
// relayfs 配置
#define SUBBUF_SIZE (256 * 1024)    // 子缓冲区大小: 256KB
#define N_SUBBUFS   8               // 子缓冲区数量: 8
// 总缓冲区大小: 2MB per CPU
```

### 2. 溢出处理策略

```c
// 当缓冲区满时的策略
enum overflow_policy {
    OVERWRITE_OLDEST,   // 覆盖最旧数据（可能丢失）
    BLOCK_WAIT,         // 阻塞等待（影响性能）
    DROP_NEWEST,        // 丢弃新数据
};
```

### 3. 同步机制

```c
// 内核侧
spinlock_t buffer_lock;
wait_queue_head_t read_wait;

// 用户侧
pthread_mutex_t data_mutex;
pthread_cond_t data_cond;
```

## 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 缓冲区溢出 | 数据丢失 | 增大缓冲区、多级缓冲 |
| 用户态进程崩溃 | 数据丢失 | 内核侧临时缓存、进程重启恢复 |
| 性能瓶颈 | 延迟增加 | 多线程、批量处理 |
| 数据顺序错乱 | 一致性问题 | 序列号排序 |
